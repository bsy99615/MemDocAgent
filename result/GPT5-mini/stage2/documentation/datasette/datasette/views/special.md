# `special.py`

## `datasette.views.special.JsonDataView` · *class*

## Summary:
A small async view class that exposes JSON data (either as a raw JSON response or rendered inside an HTML page) produced by a provided synchronous data callback and guarded by the instance-level "view-instance" permission.

## Description:
JsonDataView is intended for registering a simple read-only endpoint that returns JSON data generated by a provided callback. It enforces the datasette instance permission "view-instance" for the requesting actor, calls the supplied data_callback (optionally passing the request), and either:
- returns a raw application/json Response when the captured URL variable "format" is truthy, or
- renders the "show_json.html" template with a pretty-printed JSON string inserted into the template context when the "format" URL variable is falsy.

Typical scenarios for instantiation:
- Exposing a generated JSON file for download or API consumption.
- Displaying the same JSON inside a web page for browser viewing (human-readable pretty-printed JSON).
This class is a focused abstraction: it ties together permission checking, invoking a synchronous data producer, JSON serialization, optional CORS headers, and choosing between raw JSON and HTML rendering.

## State:
- name (class attribute): str = "json_data"
  - Identifier used when registering or referencing this view type.
- ds (instance attribute): object
  - The datasette instance passed into __init__. Expected to provide:
    - an awaitable method ensure_permissions(actor, permissions_list)
    - a boolean attribute cors indicating whether to add CORS headers
  - Invariant: ds is not None and must implement the methods/attributes above for get() to function.
- filename (instance attribute): str
  - A label or filename to pass into the template context when rendering the HTML view.
  - Constraint: any string usable by the template; not validated by this class.
- data_callback (instance attribute): callable
  - A synchronous callable invoked to obtain the JSON-serializable data to return.
  - If needs_request is False: callable should accept zero arguments.
  - If needs_request is True: callable should accept a single argument (the request).
  - Return value: any object acceptable to json.dumps (i.e., JSON-serializable). If json.dumps fails, the exception propagates.
  - Invariant: must be callable and must not be an async coroutine function (the class calls it synchronously).
- needs_request (instance attribute): bool (default False)
  - If True, the request object is passed to data_callback(request). Otherwise data_callback() is called with no args.
  - Constraint: matches the callable signature of data_callback.

Class invariants:
- After __init__, ds, filename, data_callback, and needs_request are set and never mutated by the class.
- get() relies on ds.ensure_permissions being awaitable and ds.cors being truthy/falsey; if those properties are not present on ds, get() will raise AttributeError at runtime.

## Lifecycle:
Creation:
- Instantiate with: JsonDataView(datasette, filename, data_callback, needs_request=False)
  - datasette: required object implementing ensure_permissions(actor, permissions_list) and exposing .cors
  - filename: a string describing the output file or shown in the HTML view
  - data_callback: synchronous callable that returns JSON-serializable data; signature depends on needs_request
  - needs_request: optional bool defaulting to False

Usage:
- The single public asynchronous method to call is get(request).
- Typical call sequence:
  1. await instance.get(request)
- Behavior of get(request):
  - Reads request.url_vars["format"] to determine response mode (raw JSON vs HTML).
  - Awaits ds.ensure_permissions(request.actor, ["view-instance"]). Any exception raised by ensure_permissions propagates to the caller.
  - Calls data_callback synchronously:
    - data_callback(request) if needs_request is True
    - data_callback() if needs_request is False
  - If format is truthy:
    - Builds headers = {}; if ds.cors is truthy, calls add_cors_headers(headers) to add CORS headers.
    - Returns a datasette.utils.asgi.Response containing json.dumps(data) as the body and content_type "application/json; charset=utf-8" (headers included).
  - If format is falsy:
    - Calls await self.render(["show_json.html"], request=request, context={ "filename": filename, "data_json": json.dumps(data, indent=4) })
    - Returns whatever self.render() returns (expected to be an awaitable that yields a Response-like result suitable for the framework).
- Destruction:
  - No explicit cleanup required by this class. It does not open resources that require closing. Relying code should manage long-lived resources referenced by the datasette instance or the data_callback.

## Raises:

- get(request):

  - Any exception raised by data_callback(...) will propagate.

## Example:
- Create a JsonDataView that returns a fixed dictionary (raw JSON mode when format URL variable is truthy):

  # When an incoming request has request.url_vars["format"] truthy:
  #   await view.get(request) -> Response("{"now":1}", content_type="application/json; charset=utf-8")

- Create a JsonDataView whose generator needs the request:

      # build dict using req (cookies, actor, query params, etc.)
      return {"user": getattr(req.actor, "name", None)}
  view2 = JsonDataView(datasette_instance, "user.json", generator, needs_request=True)
  # The framework should await view2.get(request) to obtain either raw JSON or an HTML rendering with pretty-printed JSON depending on request.url_vars["format"].

### `datasette.views.special.JsonDataView.get` · *method*

## Summary:
Asynchronously handle an HTTP GET for this JsonDataView: enforce the "view-instance" permission, call the configured data provider to obtain a JSON-serializable object, and return either a raw JSON Response (for format requests) or an HTML page that displays the pretty-printed JSON. Does not mutate the view object.

## Description:
This async GET handler is intended to be invoked by the web server/ASGI routing layer when a client requests the endpoint served by an instance of JsonDataView. Typical callers:
- An ASGI route dispatcher or a BaseView dispatch method that forwards incoming GET requests to this method.
- Any test harness that simulates an HTTP GET by awaiting view_instance.get(request).

Lifecycle/context:
- It runs during request handling: validate permissions first, obtain data from a supplied callback, then choose how to render that data depending on the request URL variables.

Why a separate method:
- The permission check, data retrieval, and two rendering modes (raw JSON vs HTML template) are cohesive request-handling steps that belong to the view object; keeping them in a dedicated method keeps routing/dispatch logic clean and centralizes view behavior for easier testing.

## Args:
    request (object): ASGI/Request-like object expected to provide at least:
        - url_vars (mapping): must contain the key "format". The truthiness of request.url_vars["format"] determines response mode.
        - actor (object): the actor whose permissions will be validated by self.ds.ensure_permissions.
    No default values.

## Returns:
    Awaitable that resolves to an ASGI Response-like object.
    - When request.url_vars["format"] is truthy:
        - Returns an instance of datasette.utils.asgi.Response containing:
            - body: json.dumps(data) (compact JSON string, no indentation)
            - content_type: "application/json; charset=utf-8"
            - headers: dict possibly augmented by add_cors_headers if self.ds.cors is true
    - When request.url_vars["format"] is falsy:
        - Returns whatever self.render(...) returns (expected to be an awaitable yielding an ASGI response that renders the template "show_json.html"). The render context includes:
            - "filename": self.filename
            - "data_json": json.dumps(data, indent=4) (pretty-printed JSON)
    Edge-case returns:
        - If json.dumps fails (e.g., data contains non-serializable objects), the method will raise the original json exception (TypeError).

## Raises:
    - Any exception raised by await self.ds.ensure_permissions(request.actor, ["view-instance"]).
        - In typical Datasette implementations this will raise a Forbidden-like exception if the actor lacks permission.
    - Any exception raised by self.data_callback(...) (propagated).
    - json.JSONEncodeError/TypeError (or underlying json.dumps exception) if the returned data is not JSON-serializable.
    - Any exception raised by self.render(...) when called for the HTML path.

## State Changes:
    Attributes READ:
        - self.ds (expects an object with method ensure_permissions(actor, permissions) and attribute cors (truthy/falsey))
        - self.needs_request (bool) to decide whether to call data_callback with the request
        - self.data_callback (callable) used to obtain the payload
        - self.filename (str) passed into template context for the HTML view
        - request.url_vars["format"] (truthiness)
        - request.actor
    Attributes WRITTEN:
        - None. This method does not modify attributes on self.

## Constraints:
    Preconditions:
        - request.url_vars must include the key "format" (any value is accepted; truthiness controls the branch).
        - self.data_callback must be a synchronous callable (the method calls it directly, not awaited). If needs_request is True, it must accept a single positional argument (the request); otherwise it must accept no arguments.
        - self.ds must implement ensure_permissions(actor, permissions) as an awaitable and have an attribute cors (boolean or truthy/falsey).
        - The object returned by data_callback must be JSON-serializable by json.dumps when used in the chosen branch.
    Postconditions:
        - No mutation to the JsonDataView instance's attributes.
        - A Response-like object has been returned (either directly constructed for raw JSON or produced by awaiting self.render for the HTML path).
        - For the HTML branch, the render context contains filename and a pretty-printed JSON string under "data_json".

## Side Effects:
    - Calls await self.ds.ensure_permissions(request.actor, ["view-instance"]) — may perform I/O, database lookups, or raise permission-related exceptions.
    - Calls self.data_callback(...) — may perform arbitrary work (I/O, DB queries) and raise exceptions; the method does not catch these.
    - Calls json.dumps on the data (CPU and memory cost; full JSON string is held in memory).
    - If self.ds.cors is truthy and format is requested, calls add_cors_headers(headers) which mutates the headers dict to add CORS headers.
    - Calls self.render(...) in the non-format branch — this will typically perform template rendering and I/O.

## `datasette.views.special.AuthTokenView` · *class*

## Summary:
Represents the HTTP view that consumes a one-time "root" auth token and, on success, signs the actor as the root user by setting a cookie and redirects to the instance home. It enforces single-use consumption of the server's root token.

## Description:
This view implements the endpoint that accepts a confirmation token (query parameter "token") and, if it exactly matches the current in-memory root token, marks that token as used and authenticates the requestor as the special "root" actor by setting a signed cookie ("ds_actor"). Typical callers/factories:
- The web framework/router that mounts Datasette's view classes and provides each view instance with access to the Datasette application state via self.ds (the Datasette instance).
- Tests or internal wiring that emulate a GET request with a query-string token.

Motivation and responsibility boundary:
- Motivation: provide a minimal, auditable single-use mechanism to convert possession of a server-generated root token into a signed root actor cookie for subsequent requests.
- Responsibility boundary: this view only validates and consumes the root token, sets the cookie, and issues a redirect. It does not implement broader authentication logic, cookie signing mechanics, or persistent storage of tokens — those responsibilities belong to the Datasette instance (self.ds) and underlying utilities (Response, Forbidden, sign()).

## State:
Attributes owned by the class (visible in source):
- name (str)
  - Value: "auth_token"
  - Purpose: view name/identifier used in routing or URL generation.
  - Invariant: immutable class-level identifier for this view.
- has_json_alternate (bool)
  - Value: False
  - Purpose: indicates whether this view exposes a JSON alternate representation.
  - Invariant: constant at class-level.

Implicit/required runtime state (expected to exist on the instance; not declared here but referenced by the implementation):
- self.ds (Datasette-like application object)
  - Required attributes and semantics used by this view:
    - _root_token (str or None)
      - Meaning: current one-time root token string or None when already consumed.
      - Valid values: non-empty string means token available; None (or other falsy) means token already consumed/unavailable.
      - Invariant: when set to None, the token must be considered used and will not authenticate again.
    - urls.instance() -> str
      - Meaning: returns the URL to redirect to after successful authentication (typically the site root).
    - sign(payload: dict, type_hint: str) -> str
      - Meaning: returns a signed string used as the cookie value for actor authentication.
  - The class does not mutate other self.ds attributes except setting self.ds._root_token = None on successful consumption.

Request-level state:
- request
  - Expected shape/behavior: an object that exposes request.args supporting .get("token") to access the query-string "token" parameter.
  - No assumption is made about other request properties.

Response-level state:
- Returns a datasette.utils.asgi.Response (redirect) with a cookie set via response.set_cookie(name, value).

Class invariants:
- After a successful get(), self.ds._root_token is set to None and remains None (preventing reuse).
- The view will never return a 200 body response; it either returns a redirect Response on success or raises Forbidden on failure.

## Lifecycle:
Creation:
- Instantiation: the view is normally instantiated by the application router/dispatcher that supplies the view instance with access to the Datasette application object as self.ds. The source requires self.ds to be present at call time; the class itself defines no explicit __init__ in source so it inherits BaseView construction semantics.

Usage:
- Main entrypoint: async get(request)
  - Expected call sequence:
    1. A GET request arrives with a query parameter "token" (e.g., GET /-/auth-token?token=ABC).
    2. The router invokes AuthTokenView.get(request).
    3. The method checks for the presence of a root token in self.ds._root_token; if missing, it raises Forbidden("Root token has already been used").
    4. If present, it compares the provided token against the stored token using secrets.compare_digest (constant-time comparison).
    5. If the comparison matches:
       - It sets self.ds._root_token = None (consumes the token).
       - Creates a redirect Response to self.ds.urls.instance().
       - Sets a cookie named "ds_actor" with value self.ds.sign({"a": {"id": "root"}}, "actor").
       - Returns the redirect Response.
    6. If the comparison fails, it raises Forbidden("Invalid token").
- There is no other public API on the view; only GET behavior is implemented.

Destruction:
- No explicit cleanup is required by the view. It does not hold open resources or implement context manager semantics. All persistent side effects are explicit (setting ds._root_token to None and setting a cookie on the Response).

This coroutine is the POST handler for the MessagesDebugView endpoint. It executes during the view's request lifecycle when a client submits the debug message form/endpoint. The method:
- awaits self.ds.ensure_permissions(request.actor, ["view-instance"]) to enforce that the requesting actor has the "view-instance" permission (permission enforcement occurs before any message is added),
- awaits request.post_vars() to obtain POSTed form fields,
- reads the "message" and "message_type" form values, applying defaults if absent,
- maps the message_type to one or more Datasette message levels and calls datasette.add_message(request, message, level) for each level,
- returns a redirect Response to the instance URL returned by self.ds.urls.instance().

This logic is separated into the POST handler because it represents the full request-handling flow for posted debug messages (permission check → parse → dispatch → redirect) and should remain distinct from GET rendering or other view behaviors.

Known callers / lifecycle:
- Invoked by the MessagesDebugView when handling an incoming HTTP POST request for the debug-messages endpoint as part of normal request dispatch for that view.

## Args:
    request (object): ASGI request-like object for the current HTTP request.
        Required attributes/methods used by this method:
            - actor (any): actor object passed to permission checks.
            - post_vars() -> awaitable[Mapping[str, str]]: coroutine that resolves to a mapping of POST form fields.
        Notes:
            - The method uses await request.post_vars(), so post_vars must be awaitable and return a mapping supporting .get(key, default).

## Behavior (POST fields and defaults):
    - message (str)
        - Description: text to be added as the message.
        - Default when missing: "" (empty string).
    - message_type (str)
        - Allowed values: "INFO", "WARNING", "ERROR", "all".
        - Default when missing or empty: "INFO".
        - Semantics:
            * "INFO", "WARNING", "ERROR": add exactly one message at the corresponding level.
            * "all": add exactly three messages — one at INFO, one at WARNING, and one at ERROR.

## Returns:
    datasette.utils.asgi.Response
        - The method returns the Response produced by Response.redirect(self.ds.urls.instance()).
        - That Response has an empty body and a Location header set to the string returned by self.ds.urls.instance().
        - Response.redirect is called with no explicit status here; its default status is 302 (standard temporary redirect) per the Response.redirect contract.

## Raises:
    - datasette.utils.asgi.Forbidden
        - Condition: raised by self.ds.ensure_permissions(request.actor, ["view-instance"]) if the actor lacks the "view-instance" permission. The permission check is awaited at the start; no messages are added if this raises.
    - AssertionError
        - Condition: if the resolved message_type (after applying the default "INFO") is not one of ("INFO", "WARNING", "ERROR", "all"). The source uses assert to enforce this, which raises AssertionError when false.
    - AttributeError / TypeError
        - Possible triggers:
            * request missing .actor or .post_vars, or post_vars not awaitable.
            * self.ds missing expected attributes (INFO/WARNING/ERROR constants, add_message, or urls).
        - These are runtime errors resulting from invalid request or view state; not explicitly handled by this method.
    - Any exception raised by datasette.add_message(request, message, level)
        - The method does not catch exceptions from add_message; they propagate to the caller.

## State Changes:
    Attributes READ:
        - self.ds: read to call ensure_permissions, to access message-level constants (self.ds.INFO, self.ds.WARNING, self.ds.ERROR), to call add_message, and to access self.ds.urls.instance().
    Attributes WRITTEN:
        - None on self (the view instance itself is not mutated).
    External state mutated:
        - Datasette's message store or related state via calls to datasette.add_message(...):
            * If message_type == "all": exactly three calls are made — add_message(..., INFO), add_message(..., WARNING), add_message(..., ERROR).
            * Otherwise: exactly one call is made using getattr(datasette, message_type) to resolve the level constant.

## Constraints:
    Preconditions:
        - request must provide .actor and an awaitable .post_vars() that returns a mapping.
        - The caller's actor must be authorized for "view-instance"; otherwise ensure_permissions will raise Forbidden and no messages will be added.
        - self.ds must expose:
            * ensure_permissions(actor, permissions: Sequence[str]) which is awaitable,
            * add_message(request, message: str, level: int),
            * INFO, WARNING, ERROR integer or enum-like constants,
            * urls.instance() -> str (returns the path used for the redirect Location header).
        - message_type must be one of the allowed strings after applying default "INFO".
    Postconditions:
        - On successful return (no exception), either one or three messages have been added to the Datasette instance as described above, and the caller receives a Response redirecting to self.ds.urls.instance().

## Side Effects:
    - Permission checking has side effects managed by self.ds.ensure_permissions (e.g., logging or audit); this method awaits that call.
    - Calls datasette.add_message(...) mutate server-side message state (in-memory or persistent), which may affect what the next rendered page displays.
    - Returns a Response object instructing the client to navigate to the instance URL. The method itself does not perform network I/O; the Response is handed back to the ASGI layer to be sent.
    - No headers or Response objects are mutated here beyond constructing and returning the redirect Response via Response.redirect; Response.redirect behavior (e.g., header mutation semantics) is governed by its own contract.
